- **Passo de Build Duplicado:** O arquivo `.github/workflows/ci.yml` possui o comando `- run: npm run build` duplicado consecutivamente no job `lint-and-test`.
- **Furo de Segurança no Deploy de Produção (Sem Restrição de Branch):** O workflow `migrate-prod.yml` não possui restrição para rodar apenas na `main` (`if: github.ref == 'refs/heads/main'`). Um desenvolvedor pode acionar o workflow a partir de uma branch arbitrária e aplicar migrações não-aprovadas diretamente no banco de produção.
- **Job de Migração Ausente no Merge para Main:** O job `db-migrate` do CI roda estritamente em `pull_request` (`if: github.event_name == 'pull_request'`). Quando ocorrer um `push` para a `main`, o banco de testes não será atualizado, podendo causar falhas de testes no pipeline da branch principal.
- **Concorrência Intermitente no Banco de CI Compartilhado:** O uso de um único banco `agenda-clubber-ci` para todos os PRs (conforme descrito no `CONTRIBUTING.md`) fará com que testes e-2-e rodando simultaneamente em PRs diferentes manipulem a mesma base de dados, gerando *flaky tests* crônicos e corrompendo as validações de CI.
- **Omissão do `DATABASE_URL` no Job de Testes:** O job `lint-and-test` no `ci.yml` possui várias variáveis de ambiente, mas omite o `DATABASE_URL`. Se o Vitest, Playwright ou mesmo o build do Next.js precisarem se conectar ao banco para gerar a build estática ou executar integrações, o pipeline irá falhar.
- **Ausência de Inicialização do Servidor para o Playwright:** O `ci.yml` executa `npx playwright test` sem um passo anterior explícito para iniciar a aplicação (ex: `npm run start`). Se o `playwright.config.ts` não possuir a opção `webServer` devidamente configurada, os testes falharão instantaneamente por erro de conexão recusada.
- **Inconsistência Perigosa entre `push` e `migrate`:** O `CONTRIBUTING.md` recomenda `npx drizzle-kit push` no desenvolvimento local. Misturar state-sync manual (`push`) com versionamento de SQL (`migrate`) corrompe o rastreamento da tabela de migrations do Drizzle quando os devs tentarem sincronizar schemas locais.
- **Validação de Input de Deploy Case-Sensitive e Silenciosa:** A verificação em `migrate-prod.yml` (`if: github.event.inputs.confirm == 'deploy'`) fará com que o deploy falhe silenciosamente (skipping job) se o usuário digitar "Deploy" com 'D' maiúsculo, causando confusão na equipe ao não fornecer um erro claro de validação.
- **Omissão de Documentação Crítica sobre RLS:** O `CONTRIBUTING.md` alega que as políticas RLS vêm das migrations, mas o fluxo local de gerar migrations (`npx drizzle-kit generate`) não cria automaticamente políticas RLS no Drizzle. O documento esconde a necessidade crítica do desenvolvedor escrever SQL manual para políticas de segurança.
- **Falta do Uso de GitHub Environments para Produção:** Em vez de usar a verificação frágil de digitação (`confirm == 'deploy'`), o `migrate-prod.yml` não utiliza os recursos nativos de `environment: production` do GitHub Actions, perdendo histórico de aprovações nativas, regras de proteção (required reviewers) e escopo isolado de secrets.
